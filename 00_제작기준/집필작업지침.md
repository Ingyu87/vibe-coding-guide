## 1. 기준 문서

집필은 두 문서를 함께 기준으로 삼는다.

- `최종본_바이브코딩_상세목차가이드.docx`: 전체 구조, 장별 목적, 핵심 주장, 넣을 내용, 피해야 할 내용, 예시 장면, 체크리스트의 기준
- `서식가이드.docx`: 문체, 어말어미, 용어 표기, 제목 체계, 강조 방식, 표와 박스, 체크리스트 표현의 기준

상세목차가이드는 "무엇을 쓸 것인가"의 기준이고, 서식가이드는 "어떤 톤과 형식으로 쓸 것인가"의 기준이다.

## 2. 전체 집필 원칙

이 자료집은 교사를 개발자로 만들기 위한 기술 매뉴얼이 아니라, 교사가 교육적 책임을 유지한 채 필요한 도구를 직접 기획하고 검토하고 활용하도록 돕는 안내서다.

글은 다음 관점을 유지한다.

- AI와 코딩을 과장하지 않는다.
- 교사의 판단, 학생의 배움, 학교 현장의 조건을 먼저 둔다.
- 기능 구현보다 교육적 필요, 성취기준, 개인정보, 배포 책임을 먼저 설명한다.
- "바로 만들기"보다 "왜, 어디까지, 어떻게 책임 있게 만들 것인가"를 먼저 묻는다.
- 금지와 불안을 앞세우기보다, 기준과 절차를 제시한다.

## 3. 독자 기준

주 독자는 코딩 경험이 많지 않지만 학교 현장의 문제를 해결하고 싶은 교사다.

따라서 문장은 다음 조건을 만족해야 한다.

- 전문용어를 쓸 때는 처음 등장하는 곳에서 짧게 풀어 쓴다.
- 기술 개념은 학교 장면, 수업 흐름, 행정업무 장면과 연결해 설명한다.
- 개발자식 표현보다 교사가 실제로 판단할 수 있는 질문으로 바꾼다.
- "따라 하면 된다"보다 "스스로 점검할 수 있다"에 가깝게 쓴다.

## 4. 문체 기준

본문은 단정한 설명체를 기본으로 한다.

- 본문 설명: `~다`, `~한다`, `~필요하다`
- 교사용 안내: `~해 주세요`, `~확인해 주세요`
- 체크리스트: `~있는가?`, `~되는가?`, `~확인했는가?`

문장은 짧게 쓴다. 한 문장에 여러 판단을 겹치지 않는다. 한 문단에는 하나의 핵심 생각만 담는다.

피해야 할 표현은 다음과 같다.

- 홍보물처럼 들뜬 표현
- 기술 매뉴얼처럼 딱딱한 명령문
- AI가 모든 문제를 해결할 것처럼 보이는 표현
- 교사의 책임을 AI나 도구에 넘기는 표현
- 근거 없이 "반드시", "최고", "완벽"처럼 단정하는 표현

## 5. 장별 작성 구조

각 장은 상세목차가이드의 장별 항목을 다음 순서로 풀어 쓴다.

1. 장의 문제의식
2. 교사가 마주하는 실제 장면
3. 핵심 개념 또는 기준
4. 적용 절차
5. 주의할 점
6. 예시
7. 체크리스트
8. 다음 장으로 이어지는 마무리

모든 장에 이 순서를 기계적으로 반복할 필요는 없지만, 독자가 "왜 필요한가 → 무엇을 판단해야 하는가 → 어떻게 적용할 것인가"의 흐름을 따라갈 수 있어야 한다.

## 6. Part별 집필 방향

### 프롤로그

독자가 이 자료집을 읽어야 하는 이유를 설명한다. AI를 잘 쓰는 기술보다 교사답게 쓰는 기준이 중요하다는 문제의식을 드러낸다. 불안 조성이나 기술 유행 소개에 머물지 않는다.

### Part 1. 총론

바이브코딩을 무조건 장려하거나 금지하지 않는다. 교사, 학생, AI가 각각 어디에 있어야 하는지 설명한다. Human in the Loop는 기술 용어가 아니라 교육적 책임의 구조로 풀어 쓴다.

### Part 2. 공통

모든 도구 개발 전에 확인해야 할 공통 기준을 다룬다. 성취기준, 개인정보, 배포 범위, 학교 환경, 유지보수, 평가 공정성, 학생 안내를 중심에 둔다.

### Part 3. 유형별 개발 가이드

수업용 도구와 행정업무용 도구를 구분한다. 각 유형은 "무엇을 만드는가"보다 "어떤 위험과 책임을 먼저 확인해야 하는가"를 앞세운다. 도구 유형별로 교사가 꼭 넣을 한 줄, 기술 조건, 체크리스트를 포함한다.

### Part 4. 적용과 그 이후

개발 이후의 운영, 공유, 수정, 폐기, 기록을 다룬다. 도구를 한 번 만드는 것으로 끝내지 않고 학교 안에서 책임 있게 관리하는 흐름을 제시한다.

## 7. 예시 작성 기준

예시는 실제 학교 장면처럼 써야 한다.

- 교사 이름, 학생 이름 등 실제 개인정보처럼 보이는 정보는 쓰지 않는다.
- 특정 상용 서비스 사용을 전제로 고정하지 않는다.
- 좋은 예시와 위험한 예시를 함께 제시할 수 있다.
- 예시는 기능 설명이 아니라 판단 기준을 보여주는 장면이어야 한다.

## 8. 체크리스트 작성 기준

체크리스트는 실행 가능한 질문으로 쓴다.

좋은 체크리스트 예시는 다음과 같다.

- 이 도구가 해결하려는 교육적 문제가 분명한가?
- 학생 개인정보를 입력하지 않아도 사용할 수 있는가?
- AI 결과를 교사가 최종 확인하는 절차가 있는가?
- 학생이 AI 결과를 그대로 제출하지 않도록 안내했는가?
- 배포 후 오류를 수정하거나 사용을 중단할 기준이 있는가?

체크리스트에는 추상적인 선언을 넣지 않는다. `AI 윤리를 지켰는가?`보다 `학생 개인정보가 외부 서비스로 전송되는지 확인했는가?`처럼 확인 가능한 문장으로 쓴다.

## 9. 사전조사·출처 활용 기준

원고 작성 전에는 `01_사전조사`와 `02_출처`를 함께 확인한다. 상세목차가이드는 장의 구조를 정하고, 사전조사와 출처 자료는 장의 근거와 현장성을 보강한다.

출처 자료는 본문을 장식하기 위한 인용이 아니라, 기준을 세우는 근거로 사용한다. 원문 내용을 길게 옮기지 않고, 자료집의 문체에 맞게 판단 기준, 예시 장면, 체크리스트, 주의사항으로 바꾸어 쓴다.

PDF/PPTX 출처는 원본 파일을 바로 읽기보다 `02_출처_MD`의 변환본을 우선 확인한다. 이미지 중심 페이지나 슬라이드는 `02_출처_MD/OCR_이미지보강_현황.md`, `00_제작기준/이미지보강_상세분석.md`, contact sheet를 함께 본다. 텍스트 추출본만 보고 원고를 쓰지 않는다.

### 9-1. 사전조사 자료의 역할

`01_사전조사` 폴더의 자료는 집필 방향을 잡는 데 사용한다.

- 브레인스토밍 문서: 유사 자료의 한계, 우리 자료집의 차별점, 핵심 집필 방향 확인
- 큰 DOCX 브레인스토밍 자료: 현장 제안, 표, 카드형 아이디어, 초안 표현 참고
- 목차 비교 메모: 최종 목차가 왜 지금 구조로 정리되었는지 확인

사전조사 자료는 원고의 직접 출처라기보다 "왜 이 장이 필요한가"를 뒷받침하는 배경 자료다. 원고에 반영할 때는 다음 방식으로 바꾼다.

- 유사 자료의 한계 → 이 장의 문제의식
- 차별화 포인트 → 핵심 주장
- 집필진 제안 → 예시 장면 또는 반복 박스
- 현장 우려 → 핵심 위험과 체크리스트

### 9-2. 출처 자료의 역할

- 해커톤 사례집: 교사 개발자 사례, 윤리 자기 점검, 현장 언어
- 디버깅 강의 자료: 오류 공유, 새 대화방 분리, 전체 코드와 오류 메시지 전달 방식
- 웹앱 기초 자료: 기획, 설계, 개발, 배포, 운영 흐름
- 학습지원 소프트웨어 심의 가이드: 개인정보, 학운위 심의, 교육자료 선정 기준
- 교육과정 CSV: 성장형 교사 개발자 연수 흐름과 차시 구성 참고

### 9-4. 출처 반영 방식

출처를 본문에 반영할 때는 다음 순서를 지킨다.

1. 해당 장의 `toc_detail.md` 항목을 먼저 확인한다.
2. 사전조사 자료에서 이 장의 문제의식과 차별점을 찾는다.
3. 출처 자료에서 현장 사례, 제도 기준, 기술 절차, 위험 요소를 찾는다.
4. PDF/PPTX 자료는 `02_출처_MD`의 변환본을 먼저 보고, 이미지 중심 자료는 contact sheet까지 확인한다.
5. 찾은 내용을 원문 인용이 아니라 자료집 문체의 설명, 표, 박스, 체크리스트로 바꾼다.
6. 점검표에 어떤 사전조사·출처 자료와 어떤 이미지 보강 자료를 참고했는지 남긴다.

본문에 출처를 반영하는 권장 형태는 다음과 같다.

- 제도 기준 → "확인해야 할 기준" 또는 체크리스트
- 사례 자료 → "예시 장면"
- 연수 자료 → "작업 순서" 또는 "개발 고려사항"
- 디버깅 자료 → "오류 대응 기준" 또는 "AI에게 전달할 정보"
- 브레인스토밍 자료 → "문제의식" 또는 "교사라면 꼭 넣을 한 줄"

### 9-5. 출처 사용 시 피할 것

- 출처 자료 문장을 길게 그대로 옮기지 않는다.
- 출처 이름만 붙이고 본문 판단과 연결하지 않는 방식은 피한다.
- 특정 도구나 서비스 사용을 근거 없이 표준처럼 쓰지 않는다.
- 사례집의 개별 사례를 모든 학교에 적용되는 일반 원칙처럼 단정하지 않는다.
- 심의 가이드 내용을 법률 자문처럼 과장하지 않는다. 학교 현장에서 확인해야 할 기준으로 설명한다.

### 9-6. 장별 점검표에 남길 항목

각 장의 점검표에는 다음 항목을 반드시 포함한다.

| 점검 항목 | 확인 내용 |
|---|---|
| 참고한 사전조사 | 어떤 브레인스토밍/조사 문서를 보았는가 |
| 참고한 출처 | 어떤 PDF, PPTX, CSV, 사례집을 보았는가 |
| 참고한 변환본 | `02_출처_MD`의 어떤 Markdown 파일을 보았는가 |
| 이미지 보강 확인 | 이미지 중심 페이지, contact sheet, PPTX 포함 이미지를 확인했는가 |
| 반영 위치 | 문제의식, 예시 장면, 핵심 위험, 체크리스트 중 어디에 반영했는가 |
| 반영 방식 | 원문 인용, 요약, 재구성, 체크리스트화 중 무엇인가 |
| 추가 확인 필요 | 원고 확정 전 더 확인해야 할 제도, 용어, 사례가 있는가 |

상세한 자료별 분석과 장별 매핑은 `사전조사_출처_분석표.md`와 `이미지보강_상세분석.md`를 기준으로 한다.

## 10. 최종 검토 기준

초안을 쓴 뒤 다음 순서로 검토한다.

1. 상세목차가이드의 장별 목적과 핵심 주장이 반영되었는가?
2. 서식가이드의 문체와 어말어미 기준을 지켰는가?
3. 사전조사 자료의 문제의식과 차별점이 반영되었는가?
4. 관련 출처 자료가 장의 기준, 예시, 체크리스트에 반영되었는가?
5. 교사의 교육적 판단이 AI 기능보다 앞에 놓였는가?
6. 개인정보, 평가, 배포, 유지보수 책임이 빠지지 않았는가?
7. 코딩 경험이 적은 교사도 읽을 수 있는 문장인가?
8. 체크리스트가 실제로 확인 가능한 질문인가?
9. 예시가 학교 현장 장면으로 이해되는가?
